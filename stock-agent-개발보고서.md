# 📝 개발 보고서 — 주식 멀티 에이전트 만들기

> 주식 멀티 에이전트 시스템을 처음부터 만들면서 거친 과정·부딪힌 문제·배운 점을 정리한 개발 기록입니다. (코딩 초보자 관점에서 작성)

## 1. 프로젝트 개요

**목표**: 한국·미국 주식을 분석하는 AI 에이전트 시스템을 만들고, 최종적으로는 AI가 모의투자로 스스로 거래해서 수익/손실을 관찰하는 것.

**만든 방식**: Claude와 대화하며 단계별로 구축. 처음엔 Antigravity IDE, 이후 Claude Code로 전환. Python + Streamlit 기반.

**결과물**: 총괄 매니저 + 11명의 전문 직원(에이전트)으로 구성된 시스템. 종목 분석부터 실제 모의계좌 연동·포트폴리오 분석·추천·매매 규칙 판단까지.

## 2. 개발 여정 (단계별)

### 1단계 — 기본 에이전트 구조 설계
총괄 매니저가 질문을 분석해 적절한 직원에게 배분하는 계층 구조를 설계. 조회·시그널·뉴스·펀더멘털·거시 등 역할별로 나눔.

### 2단계 — 데이터 툴링 구축
시세·지표·재무·뉴스 데이터를 가져오는 함수들 구현. 한국·미국 양쪽 지원.

### 3단계 — 웹 UI (Streamlit)
채팅 형태의 웹앱. 조직도 사이드바, 대화 맥락 기억, 면책 고지.

### 4단계 — 고급 기능
급변동 원인 추적(종목 고유/섹터/시장 구분), 병렬 처리, 종목 발굴(스크리닝).

### 5단계 — 실제 모의계좌 연동 (KIS API)
한국투자증권 모의투자 계좌를 API로 읽어와, 실제 보유종목을 포트폴리오 직원이 분석.

### 6단계 — 추천 + 벤치마크
포트폴리오를 보고 부족한 섹터를 채울 종목 추천, 그리고 "내 수익률 vs 지수" 비교 기능.

### 7단계 — 자동매매 1단계: 주문 기능 + 사람 승인
처음으로 "읽기"를 넘어 "쓰기(주문)"를 만들었다. place_kis_order(모의 주문 API + 금액상한·모의계좌 안전장치 + dry_run) → 실제 1주 주문 접수까지 테스트. 그 다음 "사람 승인" 한 겹을 씌움(y/n 받아 실행). 즉, AI는 제안만, 실행 결정은 사람.

### 8단계 — 자동매매 2단계: 거래 규칙
"언제 사고 파느냐"를 코드 규칙으로. 매도 규칙(손절 -10% / 익절 +20%) + 매수 규칙 두 가지(A: 점수집중, B: 분산채우기)를 만들어 비교. 모두 "판단만 하는 dry_run". 규칙을 AI가 아니라 코드로 둔 것이 핵심 — 예측 가능하고 토큰도 안 든다.

### 9단계 — 자동매매 3단계 준비: 가드레일 + 로그 구분
실제 자동 실행 전에 안전장치를 먼저 만들었다. check_guardrails(거래시간·한도·비중·횟수·kill switch 검사) + log_trade(A/B/MANUAL 딱지로 거래 기록). 모두 실제 주문 없이 "검사/기록만". 마지막 "조립(자동 실행)"은 데이터가 쌓인 뒤로 미뤄둠.

### 10단계 — 편의성 다듬기 (관찰이 편해지도록)
데이터 쌓는 단계가 편하도록 사용성을 다듬었다:

- **규칙 웹앱 연동**: 매수 규칙 A/B와 매도 규칙을 웹앱에서 자연어로 볼 수 있게 연결(trading_rule_employee 추가). 판단만, 실제 주문은 안 나감.
- **자주 쓰는 질문 버튼**: 사이드바에 버튼 6개(포트폴리오 어때?, 종목 추천, 벤치마크 요약, 손절 대상, 규칙 A/B 비교, 거시환경) — 클릭 한 번으로 질문.
- **KIS 에러 안내 개선**: 연결 실패 시 현재 시각을 보고 "지금은 서버 점검 시간일 수 있음"을 안내 (주말·새벽 에러에 당황 안 하도록).
- **벤치마크 짧은 요약**: 기록용으로 수치만 딱 뽑아 보여주기(새 기능 없이 질문 방식으로 해결).

> 교훈: "안 만들어도 될 것"과 "만들 것"을 구분. 벤치마크 요약은 새 함수 없이 질문만으로 해결됐다 — 속도 최적화 때처럼 "불필요한 개발"을 피함.

### 11단계 — 실제 사용에서 발견한 약점 고치기
쓰다가 이상한 걸 발견해 고쳤다. **코드만 읽어서는 못 찾았을 문제들**이다.

**발견 1 — 추천이 추세를 안 봄**

사례: 추천 직원이 HD현대중공업을 "Industrials 분산에 좋다"고 추천. 근데 다시 물어보니 수주 불발 악재로 MA20·MA60을 -17~19% 하회하는 강한 하락 추세였다. 추천 직원은 실적·성장 점수와 섹터 공백만 보고, 주가 흐름을 전혀 안 봤던 것.

→ 해결: 추천 종목마다 get_indicators로 추세를 확인해 "⚠️ 하락 추세 — 진입 타이밍 주의" 경고를 붙임. **단, 추천에서 빼지는 않음** — 하락 추세가 "싼 기회"일 수도 있고, 판단은 사람이 하니까.

**발견 2 — 직원 간 모순을 아무도 안 잡음**

추천 직원은 "사라", 시그널 직원은 "하락 추세다" — 서로 모순되는데 지적하는 사람이 없었다.

→ 해결: 리스크 검수 직원의 역할 확장. 기존(숫자 오류·과장·환각 검증)에 더해 ①직원 간 모순 ②한 직원 안의 자기모순 ③빠진 관점까지 점검. 경고는 핵심 1~3개만 간결하게.

> **교훈 1**: 진짜 약점은 코드 분석이 아니라 **실제로 써보면서** 나온다. "사라더니 왜 사지 말래?"라는 사용자의 의심이 출발점이었다.
> **교훈 2**: 새 직원을 만드는 것보다 기존 직원의 역할을 확장하는 게 빠르다. 검수류 직원은 "남의 결과를 받아" 일하므로 병렬로 못 돌리고, 추가하면 호출이 하나 늘어 느려진다.

**발견 3 — 같은 문제가 다른 곳에도 있었다**

추천 직원을 고치고 보니, **매수 규칙 A/B와 매도 규칙도 똑같이 추세를 안 보고 있었다.** 점수·수익률만 보고 판단했던 것. 이 규칙들은 나중에 자동매매에서 **사람 확인 없이** 돌 것이므로 더 위험하다.

→ 해결: 두 규칙에도 추세 정보를 붙임.
- **매수 규칙 A/B**: 후보 종목마다 `trend_warning` (하락 추세 / 모멘텀 약세 / 과매도)
- **매도 규칙**: 손절·익절 후보마다 `trend_note` (반등 신호 / 과매도 / 추세 개선 / 하락 지속)
- 둘 다 **후보에서 빼지 않고 정보만 추가**. 나중에 자동매매에서 "하락 추세면 매수 보류", "반등 신호면 손절 보류" 같은 판단을 넣을 수 있게 재료를 준비한 것.

**손절이 왜 더 신중해야 하는가 — 실제 사례**

SK하이닉스가 -20%로 손절 후보 1위였다. 근데 며칠 뒤 **-5.84%까지 회복**했다. 그때 규칙대로 기계적으로 팔았다면 -20%를 확정하고 회복을 놓쳤을 것. 매수는 안 사면 그만이지만, **손절은 팔면 손실이 확정**되고 되돌릴 수 없다.

> **교훈 3**: 규칙은 항상 옳지 않다. 특히 시장 전체가 빠진 구간에서 기계적 손절은 "바닥에서 파는" 것일 수 있다. 그래서 규칙에 추세 정보를 붙이되, 최종 판단은 여전히 사람이 하도록 남겨둔다.

**발견 4 — 거래량을 가지고만 있고 안 쓰고 있었다**

시그널 직원은 MA·RSI·MACD만 보고 **거래량은 무시**하고 있었다. 그런데 거래량은 "그 움직임에 힘이 있는지"를 알려준다 — 같은 5% 하락이라도 거래량 폭증이면 "큰 악재", 거래량 부진이면 "단순 관심 부족"으로 의미가 완전히 다르다.

→ 해결: get_indicators에 `volume`, `volume_ma20`, `volume_ratio`(오늘 거래량 ÷ 20일 평균) 추가. 시그널 직원이 가격 방향과 묶어서 해석하도록 확장.
- 예시(삼성전자 실측): 하락 추세 + 거래량 0.64배(부진) → "관망 분위기, 대규모 매도세는 아님"
- 핵심은 **절대량이 아니라 `volume_ratio`(평소 대비 몇 배)**. 종목마다 규모가 달라 절대량 비교는 무의미하다.
- 비용 0: `_fetch_ohlcv`가 이미 volume을 가져오고 있었다. **데이터는 있는데 활용만 안 했던 것.**

> **교훈 4**: 새 데이터를 구하기 전에, **이미 가지고 있는 데이터를 다 쓰고 있는지** 보라. 거래량은 처음부터 있었는데 몇 주간 무시되고 있었다.

**발견 5 — 실적 발표 임박을 아무도 안 알려줌**

실적 발표 직전에는 주가 변동이 큰데, 시스템이 "이 종목 곧 실적 발표다"를 전혀 모르고 있었다.

→ 먼저 **진단부터**: 미국은 yfinance에서 다음 발표 예정일을 줄 수 있음(AAPL 7/31, NVDA 8/27 등 실측). 한국은 DART에 사전 예정일 필드가 아예 없고 발표 당일/직전에만 공시가 뜸 → 구조적으로 미리 알 수 없음.

→ 해결(방법 A): get_fundamentals에 `next_earnings_date` 추가.
- 미국: 실제 예정일 표시. 추정치면 "(추정치)" 병기. 이미 쓰던 yf.Ticker 객체 재사용이라 추가 비용 거의 없음.
- 한국: "확인 불가 (DART는 사전 예정일 미제공)" 고정. 과거 패턴 추정은 오해 위험이라 안 함 — 애매한 추정보다 "확인 불가"가 정직.
- 펀더멘털 직원: 발표 7일 이내면 "⚠️ N일 후 실적 발표 예정 — 발표 전후 변동성 주의" 경고.
- **부수 발견(버그)**: AI 모델은 "오늘이 며칠인지" 모른다. 그래서 임박 계산이 항상 틀렸음 → 호출 시점의 실제 날짜를 프롬프트에 주입해 해결.

> **교훈 5**: 반쪽짜리 정직함이 엉터리 전체보다 낫다. 한국을 억지로 추정하지 않고 "확인 불가"로 밝힐 것 — 환각 방지 원칙과 같다. 그리고 AI는 "오늘"을 모른다는 것도 배웠다 — 사람한텐 당연한 게 AI한텐 당연하지 않다.

### 12단계 — 전체 코드 리뷰: 조립 전에 지뢰 제거 (2026-07-13)

0단계 데이터가 쌓이는 동안, 조립(자동 실행) 전에 코드 전체를 Claude와 함께 리뷰했다. 고칠 것을 심각도로 나눠(🔴 즉시 / 🟡 조립 전) 6건을 발견·수정.

**🔴 즉시 고친 것 3건**

- **테스트가 실험 데이터를 지우는 지뢰**: tools.py의 테스트 블록이 실행될 때마다 실제 trade_log.json을 삭제하고 있었다. 지금은 테스트 데이터뿐이라 무사했지만, 몇 주치 A/B 실험 기록이 쌓인 뒤 무심코 `python tools.py` 한 번이면 **전부 날아갈 뻔했다.** → 테스트는 임시 파일(trade_log_TEST.json)에서만 돌고, try/finally로 예외가 나도 원래 경로로 복구되게 격리.
- **파일 경로가 실행 위치 따라 달라짐**: 로그·토큰 캐시가 상대경로라, 터미널을 어디서 열었느냐에 따라 다른 곳에 파일이 생겨 데이터가 흩어질 수 있었다. → `Path(__file__)` 기준 절대경로로 고정.
- **웹앱의 유령 직원 2명**: 매니저에는 직원 11명이 등록됐는데 app.py 라벨·조직도에는 9명뿐 — 추천·매매규칙 직원을 부르면 화면에 raw 함수명이 그대로 노출. → 라벨·상세·조직도 카드 추가로 11명 완전 일치.

**🟡 조립 전에 고친 것 3건**

- **가드레일 KST 고정 + 주말 검사**: check_guardrails가 로컬 시간을 쓰고 있어 나중에 클라우드(UTC)로 옮기면 거래시간 검사가 9시간 틀어질 뻔. 요일 검사도 없어 토요일 오전도 "통과"였다. → 항상 KST로 계산 + 거래일 검사 추가 (미국장은 토요일 새벽 = 금요일 밤 세션이므로 전날 요일로 판단).
- **오늘 누적치 자동 집계 부품**: 가드레일이 요구하는 "오늘 누적 금액·횟수"를 손으로 계산해 넘겨야 했다 — 조립 때 실수하면 가드레일이 무력화된다. → get_auto_trade_stats_today() 추가: 로그에서 매수만 합산, 미국 종목은 환율 환산(환율 조회 실패 시 근사치 대신 에러 — 가드레일 입력이 부정확하면 안 되므로).
- **매니저 중복 제거의 부작용**: "같은 직원 두 번 금지"가 이름만 보고 판단해서, "삼성전자 뉴스 보고 SK하이닉스 뉴스도"처럼 정당한 재호출까지 잘랐다. → 판단 기준을 "이름 + 지시 내용"으로 바꿔, 지시가 다르면 같은 직원도 허용.

**작업 방식 — Claude(설계) + Claude Code(실행) 분업**

채팅 Claude가 문제 진단과 **지시문 작성**을 맡고, Claude Code가 실행을 맡았다. 지시문에는 ①문제 설명 ②수정 요구사항 ③**완료 조건(직접 검증할 것)** ④"git diff 보여주고 커밋하지 마"를 넣었다. 사람은 diff를 눈으로 검토한 뒤 직접 커밋. 중간에 API 서버 오류로 작업이 끊기기도 했지만, 시작 전 git 상태를 깨끗하게 해둔 덕에 어디까지 됐는지 diff로 확인하고 이어서 완료할 수 있었다.

> **교훈 6**: 최악의 버그는 기능이 안 되는 게 아니라 **데이터를 지우는 코드**다. 기능 버그는 다시 고치면 되지만, 몇 주치 실험 데이터는 되돌릴 수 없다. 테스트는 반드시 실제 데이터와 격리할 것.

> **교훈 7**: AI에게 일을 시킬 때는 **완료 조건(검증 방법)까지 지시문에 넣어라.** "고쳐줘"가 아니라 "고치고, 이렇게 증명해줘". 그리고 AI의 "완료했습니다" 보고를 그대로 믿지 말고 diff는 사람이 직접 본다.

> **교훈 8**: /clear 후에도 통하는 **자기완결형 지시문**을 써라. 이전 대화를 몰라도 문제·수정·검증이 전부 담긴 지시문은 세션이 끊겨도, 새 세션에서도 한 번에 처리된다.

**토큰 사용량의 중요성을 깨달았다**

이번 작업을 하며 "토큰 = 돈이자 속도"라는 게 체감됐다. AI는 매 요청마다 컨텍스트(대화 맥락) 전체를 다시 읽기 때문에, 긴 세션일수록 같은 질문도 점점 비싸진다.

- **세션 관리**: 새 작업은 /clear로 맥락을 비우고 시작. 대신 지시문을 자기완결형으로 써서 왔다갔다 재설명하는 토큰을 없앤다.
- **시스템 설계도 토큰 문제다**: 매니저의 중복 직원 호출 제거 = 곧 토큰 절약(같은 발굴을 두 번 돌면 시간·비용 2배). 매매 규칙을 AI가 아닌 코드로 둔 것(8단계)도 토큰 0원 판단이라 가능했던 것.
- **아직 남은 것**: 웹앱은 대화가 길어질수록 과거 기록 전체를 매 질문마다 매니저에게 넘긴다 — 오래 쓰면 비용이 누적 증가. 최근 N개만 넘기는 개선이 후보로 남아 있다.

> **교훈 9**: 토큰 절약은 짠돌이 짓이 아니라 설계 문제다. "AI에게 무엇을 읽힐 것인가"를 정하는 것 자체가 개발의 일부이며, 좋은 구조(역할 분리·중복 제거·코드 규칙)는 대체로 토큰도 아낀다.

## 3. 부딪힌 문제와 해결

가장 많이 배운 부분. "문제 → 원인 → 해결"을 정리.

### 문제 1: 한국 뉴스 크롤링 실패
- 증상: 네이버 크롤링이 0건 반환
- 해결: 공식 API인 DART 전자공시 + 구글뉴스 RSS로 교체. **교훈: 크롤링보다 공식 API가 안정적.**

### 문제 2: 무료 데이터 소스의 간헐적 실패
- 증상: 일부 종목 조회가 랜덤하게 실패 (특히 미국 재무)
- 해결: **캐싱 + 재시도 + 부분 성공** 3종 세트. 10개 중 6개만 와도 그걸로 분석 진행.
- **교훈: 무료 소스는 실패율 0%로 못 만든다. "안 실패"가 아니라 "잘 대응"이 목표.**

### 문제 3: 속도 — 복합 질문이 너무 느림
- 증상: 종목 발굴이 703개 전체를 돌아 6분 소요
- 해결: 추천용은 시총 상위만(KR 50, US 100) 스크리닝 + 캐시. **6분 → 13초(약 24배).**
- **교훈: "전체를 다 볼 필요"와 "빠르게 쓸 만한 결과"는 다르다. 목적에 맞게 범위를 조절.**

### 문제 4: 매니저가 직원을 중복·과다 호출
- 증상: "추천해줘" 한 번에 직원을 5번 부르고, 무거운 발굴까지 중복 → 5분 낭비
- 해결: 프롬프트 설득 + **코드로 강제 차단** 두 겹. 같은 직원 중복 호출 제거.
- **교훈: 프롬프트(말로 부탁)만으로는 LLM이 가끔 안 듣는다. 중요한 건 코드로도 막는다.**

### 문제 5: 환율 때문에 손익이 헷갈린 것처럼 보임
- 증상: 주식은 손실인데 총자산은 5억 넘음
- 원인: 현금이 82% 그대로 있고, 미국 주식은 달러→원화 환율 효과로 부풀려 보임
- **교훈: 외화가 섞이면 "주식 손익(달러)"과 "원화 환산 총자산"을 분리해서 봐야 한다.**

### 문제 6: 주말·심야 KIS 서버 점검으로 계좌 조회 실패
- 증상: "내 포트폴리오 분석해줘"를 했는데 KIS 연결 오류 — 계좌를 못 읽음 (주말 밤에 발생)
- 원인: KIS 모의투자 서버는 보통 자정~오전 7시 사이에 점검. 특히 주말 밤에 점검이 잦음. 서버가 응답을 안 하니 직원도 계좌 데이터를 못 받아 분석 불가.
- 해결: 코드 문제 아님. 평일 낮(장 시간 09:00~15:30)에 다시 하면 정상 작동.
- **교훈: 외부 API는 "점검 시간"이 있다. 밤·새벽·주말에 막힐 수 있으니, 에러가 나면 "코드 문제"보다 "시간 문제"를 먼저 의심. → 자동매매(특히 미국장, 한국 새벽)를 만들 땐 "서버 점검 시간 대응"을 꼭 넣어야 함.**

### 문제 7: 매매정지 구간의 0값이 지표를 왜곡
- 계기: 예전에 주식 분할로 가격이 변동된 걸 남한테 들어 알았다. "우리 시스템은 분할을 제대로 처리하나?" 확인해보기로 함.
- **분할 자체는 안전했다**: NVDA 10:1(2024-06), 에코프로 5:1(2024-04) 모두 테스트 — 데이터 소스(네이버·야후)가 과거 가격을 분할에 맞춰 이미 재계산해서 준다. 지표·급변동 알림·손절 규칙이 분할 때문에 헛다리 짚을 일은 없음.
- **그런데 따로 문제를 발견**: 한국 종목은 액면분할 시 2~3주 매매정지가 걸리는데, 그 기간 데이터가 `open/high/low=0, volume=0`으로 채워져 온다. 이 0값 행이 MA20·RSI·MACD 계산에 섞여 지표를 왜곡.
- **실제 왜곡 정도**: 에코프로 매매정지 구간(12개 행) 포함 시 MA20이 104,238 → 필터링 후 116,766 (약 **11% 차이**). RSI도 40.8 → 35.6.
- 해결: `_filter_halted_rows()` 추가 — volume이 0/결측이거나 open/high/low 중 하나라도 0인 행을 제거. `_fetch_ohlcv`가 get_quote·get_indicators 공통 통로라 한 곳만 고쳐도 둘 다 적용됨. 삼성전자·AAPL 같은 일반 종목은 제거된 행 0개로 지표 불변.
- **교훈 1: 고치기 전에 확인하라.** 확인 없이 "분할 처리 코드 넣자"고 했으면, 이미 잘 되는 걸 건드려 망가뜨렸을 수도 있고 진짜 문제(0값)는 못 찾았을 것.
- **교훈 2: 데이터는 항상 깔끔하지 않다.** 무료 소스 실패→캐싱/재시도/부분성공, 섹터 비교→NaN 제거, 그리고 이번 매매정지→0값 제거. "데이터가 완벽하지 않다"는 걸 전제로 다뤄야 한다.

## 4. 운영하며 익힌 팁

- **코드 바꾸면 웹앱 재시작**: Claude Code로 고친 뒤 웹앱을 껐다 켜야 반영됨. 안 하면 옛 코드가 돌아 여러 번 혼란.
- **주말 KRX 점검**: 주말엔 거래소·증권사 시스템 점검이 잦아 한국 데이터가 느려질 수 있음. 코드 문제 아님.
- **GitHub 백업**: 작업 후마다 커밋으로 저장점 남기기. 키(.env)는 절대 올리지 않음.
- **한도 관리**: 긴 세션은 비싸다. 새 작업은 /clear로 맥락 비우면 한도 절약.

## 6. 배운 것 (핵심)

### 기술적으로
- 멀티 에이전트는 역할을 나누고 매니저가 조율하는 구조가 효과적
- 무료 데이터 소스는 캐싱·재시도·부분성공으로 감싸야 함
- LLM은 프롬프트만으로 안 되고, 중요한 건 코드로 강제해야 함
- 속도와 정보량은 트레이드오프 — 모든 걸 빨리 만들 필요 없음
- **토큰은 돈이자 속도** — AI는 매 요청마다 맥락 전체를 다시 읽는다. /clear + 자기완결형 지시문, 중복 호출 제거, 규칙의 코드화가 전부 토큰 절약으로 이어진다
- 최악의 버그는 기능 고장이 아니라 **데이터를 지우는 코드** — 테스트는 실제 데이터와 반드시 격리

### 투자에 대해
- **LLM은 좋은 트레이더가 아니다.** 분석은 잘하지만 "언제 사고 파는지"는 다른 영역.
- **단기 성과는 운.** 며칠 수익률로 실력을 판단할 수 없다.
- **벤치마크 비교가 핵심.** 수익률만 보면 "시장이 안 좋았나" 싶지만, 지수와 비교해야 "내 선택이 나았나"를 안다.
- **분산의 가치.** 한 종목이 폭락해도(SPCX -18%) 전체는 -2~3%에 그치게 해주는 방어막.
- **레버리지 상품의 위험.** NVDL(2배 레버리지)을 모의로 겪으며 변동성이 얼마나 큰지 체감.

### 태도에 대해
- 손실이 난 게 "처참"이 아니라, **모의로 안전하게 배운 수업료(가짜)**. 진짜 돈이었으면 아팠을 교훈을 공짜로 얻은 것.
- 서두르지 않고 단계를 밟는 게 핵심. "빨리 자동화"가 아니라 "안전하게 관찰·학습".

## 7. 다음 단계

자동매매 로드맵(별도 문서) 참고. 자동매매 **부품은 거의 다 만들어둔 상태**이다: 주문 기능·사람 승인·매도 규칙·매수 규칙 A/B·가드레일·로그 구분 — 전부 "판단/검사만 하고 실제 실행은 안 하는" 안전한 상태.

남은 건 두 가지: (1) **데이터** — 0단계에서 추천이 지수보다 나은지 2~4주 기록을 쌓는 것. (2) **조립** — 데이터가 충분해지면 만들어둔 부품들을 이어 "정해진 시간에 스스로 도는 자동 실행 프로그램"을 만드는 것.

2026-07-15에 auto_trader.py v3의 dry-run 스모크 테스트를 통과했다(규칙 A, 국내장, 유니버스 5종목). 결과는 후보 1건(SK스퀘어)이 선정됐으나 건당 예산 50만 원 < 1주 가격 141.8만 원으로 SKIPPED — 0주 주문을 시도하지 않고 건너뛰는 방지 로직이 정상 작동했다. 검증 결과 실주문 0건, dry-run 로그(auto_trader_dryrun_log.json)만 9→10건 갱신됐고, trade_log.json은 미존재 상태 그대로 유지됐다. 구조적 메모 하나: place_kis_order의 1회 100만 원 상한 때문에 1주 가격이 100만 원을 넘는 고가주는 자동매매 규칙이 애초에 매수할 수 없다 — 추후 A/B 성과를 해석할 때 "고가주가 유니버스에서 사실상 제외된다"는 점을 감안해야 한다.

> 현재는 0단계(추천 관찰) 진행 중. 부품은 다 준비됐지만, 실제 자동 실행은 데이터가 쌓이기 전까지 서두르지 않는다.
