# CLAUDE.md

이 파일은 Claude Code가 이 프로젝트에서 작업할 때 따라야 할 지침이다.

## 프로젝트 개요

한국·미국 주식을 다루는 **멀티 에이전트 시스템**. 사용자가 자연어로 질문하면 총괄 매니저가 적절한 직원(에이전트)에게 일을 배분하고, 각 직원의 보고를 종합해 균형 잡힌 답변을 제공한다.

- **범위**: 정보 조회 + 분석 + 리스크 검수까지. **실매매는 구현하지 않는다.**
- **성격**: 모든 의견은 투자 권유가 아닌 참고용. 답변에 항상 이를 명시한다.
- **금지**: 미래 주가 예측("3개월 뒤 얼마") 같은 점쟁이식 답변은 하지 않는다.

## 아키텍처

계층적 도구 호출(tool use) 구조다.

- 매니저가 **직원들을 도구처럼** 호출한다.
- 각 직원은 **자기 전용 데이터 함수만** 도구로 갖는다.
- 직원 간 역할·도구를 섞지 않는다. (조회 직원은 추천하지 않고, 시그널 직원은 사실 조회 전담 도구를 갖지 않는다.)

```
사용자 → 총괄 매니저 → 직원들 → 데이터 함수
  직원: 조회 / 시그널 / 뉴스 / 펀더멘털 / 거시 / 포트폴리오 / 비교 / 리스크검수
```

## 직원(에이전트) 명세

팀은 세 축으로 구성된다: **종목 안을 보는 직원들** + **종목 밖(환경)을 보는 직원** + **품질을 검증하는 직원**.

| 직원 | 역할 | 도구 | 원칙 |
|------|------|------|------|
| 총괄 매니저 | 질문 분석·일 배분·종합 | 직원 호출 | **질문에 꼭 필요한 직원만** 선별 호출 |
| 조회 직원 | 현재가·등락률 등 사실 | `get_quote` | 추천 금지, 사실만 |
| 시그널 직원 | 기술적 분석 (MA, RSI, MACD) | `get_indicators` | 단기 신호, 참고 의견 명시 |
| 뉴스 직원 | 뉴스·공시 요약, 호재/악재 분류 | `search_news` | 출처 명시, 사실과 해석 구분 |
| 펀더멘털 직원 | 재무 분석 (PER, PBR, 영업이익, 부채비율) | `get_fundamentals` | 중장기 근거 |
| 거시 직원 | 금리·환율·유가·지수·섹터 등 환경 분석 | `get_macro` | 개별 종목이 아닌 "판 전체" 시각 |
| 포트폴리오 직원 | 보유 종목 분산·섹터 쏠림 | `get_portfolio_analysis` | 전체 관점 |
| 비교 직원 | 종목 간 상대 평가 | 타 직원 도구 조합 | A vs B 정리 |
| 리스크 검수 직원 | 반대 시각, 과열·변동성 경고 **+ 사실 정합성·환각 점검** | 타 직원 결과를 입력받아 검증 | 쏠린 답변에 제동, 숫자 충돌·없는 사실 적발 |
| 추천 직원 | 포트폴리오 진단 → 공백 섹터 발굴 → 매수 후보 제안 | `get_kis_balance`, `get_portfolio_analysis`, `screen_stocks` | 단정 금지, 근거+위험 필수, 매도 권유 없음, 실행은 사람이 직접 |

**거시 직원**: 다른 직원이 전부 개별 종목 안만 보는 사각지대를 메운다. 종목이 좋아도 섹터·매크로가 나쁘면 답이 달라지므로, 매니저는 매수/관망 판단이 필요한 질문에서 거시 직원을 함께 부르는 것이 좋다.

**리스크 검수 직원**: 외부 데이터가 아니라 다른 직원의 결과를 입력받아 검증한다. 반대 시각 제시뿐 아니라 **직원 간 숫자가 충돌하거나 근거 없는 주장(환각)이 있는지** 점검하는 품질 감독 역할도 겸한다. 매니저는 분석 직원들을 먼저 부른 뒤 **마지막에** 리스크 검수를 호출한다.

## 기술 스택

- 언어: Python 3.10+
- 가격·지표: finance-datareader, ta, pandas
- 재무: finance-datareader / yfinance
- 거시: finance-datareader (지수·환율·금리 등)
- 뉴스: yfinance.news(미국) + 네이버 금융·DART 공시(한국). 나중에 NewsAPI/Finnhub로 교체 가능.
- LLM: anthropic (Claude API)

## 파일 구조

```
stock-agent/
├── tools.py        # 데이터·지표·뉴스·재무·거시 함수
├── employees.py    # 직원들 (각자 시스템 프롬프트 + 도구)
├── manager.py      # 총괄 매니저 + 실행 진입점
└── .env            # API 키 (절대 커밋하지 않음)
```

직원이 많으므로 `employees/` 디렉터리로 직원별 파일 분리를 권장.

## 코딩 규칙

- 데이터 함수는 항상 **dict**를 반환하고, 실패 시 `{"error": "..."}` 형태로 반환한다.
- 한국 종목은 6자리 코드(예: `005930`), 미국은 티커(예: `AAPL`)로 받는다. 두 시장을 모두 처리한다.
- 도구 함수 인터페이스(시그니처)는 안정적으로 유지한다. 데이터 소스를 바꿔도 함수 이름·인자는 그대로 둔다 (특히 `search_news`).
- LLM 모델명은 하드코딩하기 전에 **현재 유효한 최신 모델명인지 확인**한다.
- API 키는 코드에 직접 쓰지 말고 환경변수/`.env`에서 읽는다.

## 검증 순서 (아래에서 위로)

1. 데이터 함수 단위 테스트 (`get_quote('005930')` 등)
2. 직원 하나씩 단독 테스트
3. 리스크 검수 직원이 다른 직원 결과를 잘 반박·점검하는지 확인
4. 매니저 전체 실행 — 일 배분·충돌 조정·종합 확인

## 비용·성능 주의

직원이 8명이라 호출이 많으면 느리고 비싸진다. 매니저 프롬프트에서 **불필요한 직원을 부르지 않도록** 한다. 예: "애플 현재가만 알려줘" → 조회 직원만 호출.

## 인증 주의

- Claude Code 로그인(개발용 구독)과 에이전트 코드가 쓰는 API 키(실행용 Console 크레딧)는 **별개**다.
- `ANTHROPIC_API_KEY`가 설정돼 있으면 Claude Code가 구독 대신 그 키로 과금될 수 있음에 유의.
