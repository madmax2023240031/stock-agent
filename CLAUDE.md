# CLAUDE.md

이 파일은 Claude Code가 이 프로젝트에서 작업할 때 **매 세션 반드시 따라야 할 지침**이다.

## 프로젝트 개요

한국·미국 주식을 다루는 **멀티 에이전트 분석 시스템 + KIS(한국투자증권) 모의투자 자동매매 실험**.
사용자가 자연어로 질문하면 총괄 매니저가 적절한 직원(에이전트)에게 일을 배분하고,
각 직원의 보고를 종합해 균형 잡힌 답변을 제공한다.
분석 시스템 위에 자동매매 실험(A/B 매수 규칙 비교)을 단계적으로 얹어가는 중이다.

- **현재 단계: 3단계 — 승인 모드** (2026-07-20 JJG 명시 승인 "스위치 전환을 승인합니다"로 전환.
  모든 주문은 사람이 터미널에서 y를 입력해야만 실행됨).
- **성격**: 모든 의견은 투자 권유가 아닌 참고용. 답변에 항상 이를 명시한다.
- **금지**: 미래 주가 예측("3개월 뒤 얼마") 같은 점쟁이식 답변은 하지 않는다.
- 로드맵·사용 설명서의 원본은 노션이다. 저장소에는 없다.

## 절대 원칙 — 3단계 (어기면 안 됨)

1. **상수 세트 단위 전환만 허용.** `PHASE`/`MASTER_ENABLE`/`DRY_RUN`/`APPROVAL_REQUIRED`는
   세트 단위로만 전환 가능(0단계 세트 또는 3단계 세트, `_assert_phase_config`가 강제).
   개별 값 변경 금지. 전환은 상수 블록 단독 커밋 + JJG 명시 승인 하에만.
   `dry_run=False` 리터럴 호출 코드는 여전히 새로 작성하지 않는다.
2. **"데이터 먼저, 서두르지 않기."** 안전장치는 관행이 아니라 **코드로 강제**한다.
3. **모의 계좌 전용.** 실전 계좌 연동 코드는 절대 작성하지 않는다.
   (`place_kis_order`는 허용 모의계좌 하드코딩 + 1회 100만 원 상한이 이미 강제돼 있다 — 완화 금지.)
4. **`trade_log.json`(A/B 실험 장부)은 절대 오염 금지.**
   - 현재 파일이 없는 것이 정상이다. 첫 실제 기록 때 `log_trade()`가 자동 생성하고,
     `get_trade_log()`는 파일이 없으면 0건으로 처리한다. **일부러 만들지 말 것. 가짜 거래로 채우지 말 것.**
   - 테스트 코드는 `trade_log.json`을 절대 건드리지 않는다. 테스트는
     `trade_log_TEST.json` 임시 파일 + `try/finally` 정리 구조를 쓴다(`tools.py` 테스트 블록 참고).
   - dry-run 기록은 `auto_trader_dryrun_log.json`에만 쓴다. 두 파일을 섞지 않는다.

├── requirements.txt

## 아키텍처

계층적 도구 호출(tool use) 구조. 매니저가 **직원들을 도구처럼** 호출하고,
각 직원은 **자기 전용 데이터 함수만** 도구로 갖는다. 직원 간 역할·도구를 섞지 않는다.

```
사용자 → 총괄 매니저 → 직원 11명 → 데이터 함수 (tools.py)
```

| 직원 | 역할 | 주요 도구 | 원칙 |
|------|------|------|------|
| 총괄 매니저 | 질문 분석·일 배분·종합 | 직원 호출 | **꼭 필요한 직원만** 선별 호출 |
| 🔍 조회 | 현재가·등락률 등 사실 | `get_quote` | 추천 금지, 사실만 |
| 📊 시그널 | 기술적 분석 (MA, RSI, MACD) | `get_indicators` | 단기 신호, 참고 의견 명시 |
| 📰 뉴스 | 뉴스·공시 요약, 호재/악재 분류 | `search_news` | 출처 명시, 사실과 해석 구분 |
| 🏢 펀더멘털 | 재무 분석 (PER, PBR 등) | `get_fundamentals` | 중장기 근거 |
| 🌍 거시 | 금리·환율·유가·지수 환경 분석 | `get_macro` | "판 전체" 시각 |
| 💼 포트폴리오 | 보유 종목 분산·섹터 쏠림 | `get_portfolio_analysis` | 전체 관점 |
| ⚖️ 비교 | 종목 간 상대 평가 | 타 직원 도구 조합 + `get_sector_comparison` | A vs B 정리 |
| 🚨 리스크검수 | 반대 시각 + 사실 정합성·환각 점검 | 타 직원 결과 입력 검증 | 매니저가 **마지막에** 호출 |
| 🔭 발굴 | 유니버스 스크리닝 | `screen_stocks`, `get_universe` | 점수는 참고용 |
| 💡 추천 | 포트폴리오 진단 → 매수 후보 제안 | `get_kis_balance`, `get_portfolio_analysis`, `screen_stocks` | 단정 금지, 실행은 사람이 직접 |
| 📏 매매규칙 | 규칙 A/B·손절/익절 판단 | `evaluate_buy_rule_A/B`, `evaluate_sell_rules` | 판단/제안만, 주문 없음 |

직원을 추가/변경하면 `app.py`의 `EMPLOYEE_LABELS`·`EMPLOYEE_DETAIL`·사이드바 조직도도 **반드시 함께** 갱신한다.

## 자동매매 부품 (tools.py) — 전부 dry_run/검사 전용

| 함수 | 역할 | 안전장치 |
|------|------|------|
| `place_kis_order` | KIS 모의 주문 (국내 6자리 전용) | `dry_run=True` 기본, 허용 모의계좌 하드코딩, 1회 ≤ 100만 원 |
| `propose_and_confirm_order` | 주문서 → 사람 y/n 승인 → 실행 | 사람 승인 필수 (1단계 부품) |
| `check_guardrails` | 주문 전 가드레일 검사 | 주말 차단, 거래시간(KR 09:00~12:00 KST / US 09:30~12:30 **뉴욕 현지시간, 서머타임 자동 반영**), 한도 3,000만·종목 20%·섹터 40%·일 5회, kill switch |
| `get_auto_trade_stats_today` | 오늘(KST) 규칙별 누적치 집계 | 읽기 전용, 환율 실패 시 근사치 없이 에러 |
| `evaluate_buy_rule_A` | 매수 규칙 A (점수 집중) | 판단/제안만 |
| `evaluate_buy_rule_B` | 매수 규칙 B (분산 채우기) | 판단/제안만 |
| `evaluate_sell_rules` | 손절 -10% / 익절 +20% 후보 분류 | 판단/제안만 |
| `log_trade` / `get_trade_log` / `summarize_trades_by_rule` | 실험 장부 기록/조회 | 태그 `A`·`B`·`MANUAL`·`SELL`만 허용 |

- 거래 태그: `A`=매수규칙A(점수집중) / `B`=매수규칙B(분산채우기) / `MANUAL`=사람 직접 / `SELL`=매도규칙 자동매도.
- `auto_trader.py`는 **조립 전용**: 위 부품을 순서대로 연결만 하고 새 매매 로직을 넣지 않는다.
  현재 3단계 세트(`PHASE=3`/`MASTER_ENABLE=True`/`DRY_RUN=False`/`APPROVAL_REQUIRED=True`).
  `dry_run=False` 리터럴은 여전히 파일 전체 0건 유지. `place_kis_order` 호출은
  `_draft_order`·`_execute_order` 2곳뿐. `APPROVAL_REQUIRED=True` 해제는 2~3일 검증 후
  별도 세션에서 JJG 명시 승인으로만.

## 실행·테스트 규칙

- 파이썬 실행은 항상 **`./venv/bin/python`** 을 쓴다 (시스템 파이썬에는 feedparser 등이 없다).
- `tools.py` 단위 테스트: `./venv/bin/python tools.py` (테스트 블록은 `trade_log_TEST.json`만 사용).
- 매니저 테스트: `./venv/bin/python test_manager.py`.
- 웹앱: `./venv/bin/python -m streamlit run app.py` — **수정 후에는 껐다 켜야 반영**된다.
- 가드레일처럼 시각 의존 코드는 `now=` 주입 인자로 테스트한다 (실제 시계에 의존하는 테스트 금지).

## 작업 워크플로 (git)

- **자동 커밋/자동 push 절대 금지.** 커밋은 사용자가 diff를 직접 검토한 뒤 직접 한다.
- 모든 작업의 마지막은 **"커밋 없이 `git diff`만 보여주기"** 로 끝낸다.
- 커밋 메시지를 제안할 때는 `git commit -m "제목" -m "본문"` 형식으로 알려준다
  (터미널에 편집기가 등록돼 있지 않아 `-m` 방식을 쓴다).
- 작업 완료 후에는 반드시 검증 단계(무엇을 실행해 무엇을 확인하는지)를 수행하고 결과를 보고한다.

## 코딩 규칙

- 데이터 함수는 항상 **dict**를 반환하고, 실패 시 `{"error": "..."}` 형태로 반환한다.
- 실패 시 근사치로 계속 진행하지 않는다 — **fail-safe**: 판단에 필요한 입력을 못 구하면 에러를 반환하고 멈춘다
  (예: 가드레일 시간대 로드 실패, 환율 조회 실패).
- 한국 종목은 6자리 코드(예: `005930`), 미국은 티커(예: `AAPL`). 두 시장을 모두 처리한다.
  단, `place_kis_order`는 국내 전용이다.
- 시간대는 `zoneinfo.ZoneInfo`로 명시한다 (KST `Asia/Seoul`, 미국장은 `America/New_York`).
  KST 고정 오프셋으로 미국장을 판정하는 코드를 만들지 않는다 (서머타임 버그).
- 파일 경로는 상대 경로 금지 — `_BASE_DIR = Path(__file__).resolve().parent` 기준 절대 경로.
- 도구 함수 인터페이스(시그니처)는 안정적으로 유지한다. 데이터 소스를 바꿔도 함수 이름·인자는 그대로.
- LLM 모델명은 하드코딩 전에 **현재 유효한 최신 모델명인지 확인**한다.
- API 키·계좌번호는 코드에 직접 쓰지 말고 `.env`에서 읽는다.

## 검증 순서 (아래에서 위로)

1. 데이터 함수 단위 테스트 (`./venv/bin/python tools.py`)
2. 직원 하나씩 단독 테스트
3. 리스크 검수 직원이 다른 직원 결과를 잘 반박·점검하는지 확인
4. 매니저 전체 실행 — 일 배분·충돌 조정·종합 확인
5. (자동매매 부품 수정 시) 가드레일 테스트 8케이스 전부 통과 확인

## 비용·성능 주의

직원이 11명이라 호출이 많으면 느리고 비싸진다. 매니저 프롬프트에서 **불필요한 직원을 부르지 않도록** 한다.
예: "애플 현재가만 알려줘" → 조회 직원만 호출. 스크리닝 계열은 특히 느리다 — 테스트 시 `universe_limit`으로 줄인다.

## 인증 주의

- Claude Code 로그인(개발용 구독)과 에이전트 코드가 쓰는 API 키(실행용 Console 크레딧)는 **별개**다.
- `ANTHROPIC_API_KEY`가 설정돼 있으면 Claude Code가 구독 대신 그 키로 과금될 수 있음에 유의.
